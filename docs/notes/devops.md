## Network / DNS â€” image build (both debian_qcow2 scripts)

- **Rule:** Both images unify on **ifupdown + standard `/etc/network/`**, configured by the `netui` whiptail TUI (`bin/netui`, shipped to `/usr/local/bin/netui`). DHCP by default; static is set via `sudo netui` (incl. its "Capture DHCP lease -> static" action). NetworkManager and ceni are NOT used. The cloud-init image (`debian_qcow2.sh`) forces cloud-init to render via the **eni** renderer (`system_info: network: renderers: ["eni"]`). The no-cloud-init image removes cloud-init and ships `/etc/network/interfaces` = `lo` + `source /etc/network/interfaces.d/*` with a netui-owned `interfaces.d/eth0` (DHCP) + seeded `/etc/network/netui/eth0/main.conf`.
- **DNS:** `systemd-resolved` is removed; **resolvconf** manages `/etc/resolv.conf` (ifupdown `dns-*` options + dhcpcd). `/etc/resolv.conf` MUST be linked to `/run/resolvconf/resolv.conf` via `virt-customize --link` â€” NOT `--run-command 'ln -sf'`, which libguestfs reverts because it swaps `/etc/resolv.conf` for appliance networking during `--run-command`/`--install` and restores the original afterwards. Verified on live VMs: Proxmox cloud-init nameservers AND DHCP nameservers both land in `/etc/resolv.conf`.
- **Context:** Supersedes the earlier rule "cloud-init image uses the NetworkManager renderer" (commit b199fa3). The user chose to unify both images on ifupdown+netui+resolvconf. cloud-init's eni renderer places `dns-nameservers` on `lo`, but resolvconf (unlike the systemd-resolved if-up hook, which exempts lo) DOES capture lo-placed DNS, so no relocation fixup is needed. See [[1781049600-cloud-init-eni-resolvconf-dns-debian]].

## Network â€” eth0 DHCP default when cloud-init has no datasource (both cloud-init images)

- **Rule:** `debian_qcow2-cloud-init.sh` AND `debian_qcow2-cloud-init-selkin.sh` **pre-seed cloud-init's own eni render path** `/etc/network/interfaces.d/50-cloud-init` with a plain IPv4 DHCP stanza (`auto eth0` / `iface eth0 inet dhcp`) at build time. When a datasource IS present, cloud-init **OVERWRITES** this exact file with the datasource's config (verified: a static IP is applied cleanly, no leftover DHCP). When no datasource is attached (ds-identify disables cloud-init) the DHCP default stays and ifupdown brings up eth0; dhcpcd handles IPv6 via RA. SINGLE file â†’ no duplicate `eth0` stanza, no extra service.
- **Context:** Debian `ds-identify` DISABLES cloud-init when no datasource is found (default policy `notfound=disabled`); genericcloud ships no static eth0, so a VM booted WITHOUT a cloud-init drive rendered nothing and eth0 stayed `qdisc noop state DOWN`. Two rejected alternatives (both live-tested): (1) force cloud-init on via `ds-identify` `notfound=enabled` â†’ ~240s EC2 IMDS probing (169.254.169.254) + cloud-init 25.1.4 hardcodes `dhcp4+dhcp6` in `generate_fallback_config()`, whose `iface eth0 inet6 dhcp` line fails (`No DHCPv6 client software found!`; Debian 13 dropped isc-dhcp-client, image uses dhcpcd) â†’ failed `networking.service`; (2) a separate static `interfaces.d/eth0` (or a `netcfg-fallback` oneshot) â†’ cloud-init writes a SEPARATE `50-cloud-init`, so WITH a datasource both files are sourced â†’ duplicate/conflicting eth0 stanza (live test: eth0 got BOTH a datasource static IP and a DHCP lease, default route via the wrong one). Pre-seeding `50-cloud-init` itself sidesteps both â€” it is the very file cloud-init overwrites. NOTE: netui owns `interfaces.d/eth0` (a DIFFERENT file); in the cloud-init images the network authority is cloud-init/datasource, not netui â€” configuring eth0 via `sudo netui` here while `50-cloud-init` exists would create two stanzas. Details: [[1781322000-cloud-init-no-datasource-eth0-dhcp-fallback]].
- **Updated:** 2026-06-13

## Boot â€” systemd-networkd-wait-online (120s hang)

- **Rule:** Mask `systemd-networkd-wait-online.service` in BOTH images. systemd-networkd is not the manager (ifupdown is), so this waiter (enabled by the genericcloud preset, default 120s timeout) blocks `network-online.target` for the full 120s. The no-cloud-init image does not pull `network-online.target` today (so it would not hang), but it is masked there too for consistency.
- **Context:** Confirmed on a live VM: `systemd-analyze blame` â†’ `2min systemd-networkd-wait-online.service`. Details: [[1781052000-networkd-wait-online-120s-boot-hang]].

## Selkin image â€” externalized payloads (services + start scripts)

- **Rule:** The Selkies builder (`debian_qcow2-selkin.sh`) keeps its service units and start scripts as real files in the repo under `linux-debian/selkin/` instead of inline heredocs. The builder stays a SINGLE downloadable file: payloads are fetched at build time from `LINUX4U_REPO` (repo raw root `raw.githubusercontent.com/nchekwa/linux4u/${LINUX4U_REF}`, ref = `LINUX4U_REF`, default `main`); the `linux-debian/selkin/` subpath is appended inside the fetch helpers. `curl` runs on the **HOST** (to `mktemp -d` BUILD_TMP, cleaned by an `EXIT` trap), then files are injected with `virt-customize --copy-in` â€” NEVER `--run-command 'wget/curl'` inside the guest, which would hit the appliance DNS-swap (see Network/DNS rule). `netui` is fetched the same way (`${LINUX4U_REPO}/bin/netui` â†’ `--copy-in /usr/local/bin`). Templated payloads use the `.tpl` suffix and are rendered with `envsubst` using a **restricted variable list** (e.g. `envsubst '${DESKTOP_USER}'`), so build-time vars are filled while runtime `${HOME}`/`${DISPLAY}` stay literal. Secrets (`SELKIES_PASSWORD`, `VNC_PASSWORD`) stay as host env vars and are never committed. Payloads SHARED with the other builders (`quick_upgrade.sh`, `99-eni.cfg`, `50-cloud-init`) now live in `linux-debian/base/` and are pulled via a `fetch_base` helper â€” see the base/ rule below; only desktop-specific payloads stay in `linux-debian/selkin/`. A live login banner `99-motd-update.tpl` â†’ `/etc/update-motd.d/99-motd-update` (pam_motd runs `/etc/update-motd.d/*` per login on Debian genericcloud â†’ `/run/motd.dynamic`; the script resolves the VM IP fresh via `hostname -I` and prints Selkies URL/login + VNC ssh-tunnel + change-password/restart commands). **Default creds** (overridable via env): Selkies web basic-auth = `selkin`/`321selkin`, VNC = password-only `321vnc`. The system/desktop user (`DESKTOP_USER`, default `user`) is left at its existing default and is NOT tied to these. `DESKTOP_USER` has **passwordless sudo** (`/etc/sudoers.d/<user>`: `NOPASSWD:ALL`, mode 0440) so admin commands (MOTD restarts, `/opt/scripts`) never prompt. Requires `gettext-base` (envsubst) on the host.
- **Context:** Replaced 6 escaped heredocs (`\${DISPLAY}`, `\"`) that were unlintable, and moved the `netui` fetch from in-guest `wget` (hardcoded `refs/heads/main`) to host `curl` + `--copy-in` (now ref-pinned via `LINUX4U_REF`). Details: [[1781308332-selkin-payload-externalization]].
- **Updated:** 2026-06-13
